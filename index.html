Sistema de trabalho

🏗️ Fluxo de Construção Profissional

Esta é a espinha dorsal do Ouro do Dev. Todo projeto deve seguir uma ordem profissional: primeiro a estrutura, depois a aparência, depois o comportamento, depois a validação, publicação e apresentação.

Nenhuma etapa avança sem aprovação da etapa anterior. Primeiro eu construo certo, depois eu evoluo com segurança.
1. Planejamento da página Definir objetivo, público, conteúdo, páginas, seções e função do projeto.
2. HTML — Construção da estrutura Criar o esqueleto semântico com head, header, nav, main, sections, articles, aside e footer.
3. Portão de aprovação HTML Validar estrutura, semântica, acessibilidade básica, organização e prontidão para CSS.
4. CSS — Construção visual Aplicar variáveis, reset, base global, layout, componentes, responsividade e acabamento visual.
5. Portão de aprovação CSS Validar visual, responsividade, contraste, proporção, leitura, DevTools e qualidade da interface.
6. JavaScript — Comportamento Adicionar interações, eventos, DOM, validações, menus, filtros, busca e funcionalidades reais.
7. Portão de aprovação JavaScript Conferir console, eventos, acessibilidade, funcionamento no celular e ausência de erros críticos.
8. GitHub e publicação Fazer commits claros, README, GitHub Pages, link publicado e organização do repositório.
9. Portfólio e carreira Salvar prints, registrar aprendizados, explicar decisões e transformar o projeto em evidência profissional.

🧱 Portão de aprovação HTML

O HTML só está pronto para receber CSS quando sua estrutura está clara, semântica, acessível e organizada. CSS não deve ser usado para esconder HTML mal construído.

  • O documento possui <!doctype html>
  • O idioma está definido com lang="pt-BR"
  • O meta viewport foi configurado
  • O title descreve corretamente o projeto
  • A meta description existe e faz sentido
  • A página possui header, main e footer
  • As seções têm ordem lógica e títulos claros
  • Imagens informativas possuem alt
  • Formulários possuem label conectado aos campos
  • Tabelas são usadas apenas para dados reais
  • Iframes possuem title
  • O HTML está indentado e legível

🎨 Portão de aprovação CSS

O CSS só está pronto quando a interface está organizada, confortável, responsiva, acessível e coerente com o objetivo do projeto.

  • O CSS está ligado corretamente no HTML
  • Foram criadas variáveis no :root
  • O reset básico foi aplicado
  • O layout usa containers e espaçamentos consistentes
  • O site foi testado em celular
  • O site foi testado em desktop
  • Cards, botões, tabelas e formulários estão confortáveis
  • O contraste está legível
  • O foco visível foi preservado
  • A tabela não estoura a tela no celular
  • O DevTools foi usado para revisar layout
  • O visual passa organização e profissionalismo

⚙️ Portão de aprovação JavaScript

JavaScript entra para adicionar comportamento, não para bagunçar a estrutura. Uma função só deve existir quando melhora a experiência, validação ou interação do projeto.

  • O JavaScript está ligado no final do HTML
  • O console não possui erro crítico
  • Botões e eventos funcionam corretamente
  • As interações funcionam no celular
  • O JS não prejudica acessibilidade
  • O código está separado em funções claras
  • O comportamento esperado foi testado manualmente

🌐 Portão final de publicação

Um projeto só vira evidência profissional quando está publicado, documentado, testado e explicável.

  • O projeto possui README completo
  • O GitHub está organizado
  • Os commits têm mensagens claras
  • O GitHub Pages ou deploy foi testado
  • O link publicado abre corretamente
  • Prints foram salvos para portfólio
  • Eu consigo explicar o projeto em entrevista

HTML5 na prática

🧱 HTML5 — Como Construir a Estrutura do Zero

Todo projeto profissional começa antes do visual. Primeiro eu preparo a pasta, crio os arquivos, monto a estrutura HTML, organizo o conteúdo e deixo tudo pronto para receber CSS com segurança.

Eu não começo pelo CSS. Eu começo pela estrutura. Uma boa interface nasce de um HTML bem planejado.

📁 Etapa 1 — Criar a pasta do projeto

A primeira decisão profissional é organizar o projeto em uma pasta clara, com nomes simples e arquivos separados por responsabilidade. 

meu-projeto/
├── index.html
├── styles/
│   └── style.css
├── scripts/
│   └── script.js
└── assets/
    └── imagens/

Essa estrutura separa o que é página, estilo, comportamento e arquivos de mídia. Isso facilita manutenção, publicação e leitura do projeto.

🧾 Etapa 2 — Criar o arquivo principal

O arquivo principal normalmente se chama index.html. Ele é a porta de entrada do site. 

touch index.html
mkdir styles scripts assets
touch styles/style.css
touch scripts/script.js

🏗️ Etapa 3 — Escrever a base obrigatória

A base HTML precisa declarar o tipo do documento, o idioma, os metadados principais e a ligação com o CSS. 

<!doctype html>
<html lang="pt-BR">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Nome do Projeto</title>
    <meta name="description" content="Descrição clara do projeto." />
    <link rel="stylesheet" href="styles/style.css" />
  </head>

  <body>
  </body>
</html>

🧠 Etapa 4 — Pensar nas seções antes de codar

Antes de preencher o body, eu defino quais blocos a página precisa ter. Isso impede que o HTML vire uma sequência aleatória de tags.

  • Qual é o tema do projeto?
  • Quem vai usar ou avaliar essa página?
  • Qual é a primeira informação importante?
  • Quais seções precisam existir?
  • A página terá imagens, tabela, formulário, vídeo ou iframe?
  • Qual será a ação principal do usuário?

🏛️ Etapa 5 — Montar a arquitetura do body

A estrutura mínima profissional do body normalmente começa com header, main e footer. 

<body>
  <header class="site-header">
    <div class="container">
      <h1>Nome do Projeto</h1>

      <nav class="site-nav" aria-label="Navegação principal">
        <ul>
          <li><a href="index.html">Início</a></li>
          <li><a href="#sobre">Sobre</a></li>
          <li><a href="#contato">Contato</a></li>
        </ul>
      </nav>
    </div>
  </header>

  <main class="page-main">
    <section class="hero"></section>
    <section class="section" id="sobre"></section>
    <section class="section" id="contato"></section>
  </main>

  <footer class="site-footer">
    <div class="container">
      <p>© 2026 Nome do Projeto.</p>
    </div>
  </footer>

  <script src="scripts/script.js"></script>
</body>

🏷️ Etapa 6 — Aplicar classes estratégicas

As classes devem preparar o CSS. Elas precisam dizer o papel do bloco, não apenas aparência temporária.

Classe Uso Por que usar
.site-header Cabeçalho Identifica o topo do site.
.container Limite interno Controla largura e respiro lateral.
.site-nav Navegação Separa o menu principal.
.page-main Conteúdo central Marca o miolo da página.
.hero Primeira dobra Dá destaque ao bloco inicial.
.section Seção comum Cria padrão para blocos internos.
.site-footer Rodapé Identifica o fechamento da página.

♿ Etapa 7 — Revisão antes do CSS

Antes de iniciar o CSS, eu reviso o HTML como se fosse entregar para outro desenvolvedor continuar.

  • A estrutura está legível
  • O conteúdo está em ordem lógica
  • As tags semânticas fazem sentido
  • A navegação possui aria-label
  • Imagens importantes terão alt
  • Formulários terão label
  • Iframes terão title
  • Classes estão preparadas para CSS
  • O HTML aberto sem CSS ainda é compreensível

🚦 Portão de liberação para CSS

Se o HTML está completo, semântico, acessível, organizado e preparado com classes estratégicas, o projeto está liberado para iniciar o CSS.
HTML5 na prática

📝 HTML5 — Conteúdo, Links, Imagens e Mídia

Depois que a estrutura do HTML está criada, eu começo a preencher a página com conteúdo. Essa etapa define textos, títulos, listas, links, imagens, vídeos, áudios e conteúdos incorporados. Tudo precisa ter sentido, ordem e acessibilidade.

Conteúdo profissional não é jogar texto e imagem na página. É organizar informação para o usuário entender, navegar e confiar.

🧠 Etapa 1 — Definir a hierarquia do conteúdo

Antes de escrever muitos elementos, eu defino o que é título principal, o que é seção, o que é explicação, o que é lista e o que é mídia.

  • Existe um título principal claro
  • As seções têm títulos próprios
  • Os parágrafos explicam ideias completas
  • As listas agrupam informações relacionadas
  • As mídias têm função real dentro da página

🔠 Etapa 2 — Títulos e parágrafos

Títulos organizam a leitura. Parágrafos explicam. Um projeto profissional não usa títulos apenas para aumentar fonte; usa títulos para criar hierarquia. 

<section class="section" id="sobre">
  <div class="container">
    <h2>Sobre o projeto</h2>
    <p>
      Este projeto apresenta informações organizadas, acessíveis e preparadas
      para receber CSS profissional.
    </p>
  </div>
</section>

✅ Regra de headings

  • h1 deve representar o assunto principal da página.
  • h2 deve abrir seções importantes.
  • h3 deve dividir partes internas de uma seção.
  • Não use título só porque ele é grande visualmente.
  • Não pule hierarquia sem motivo.

📋 Etapa 3 — Listas

Listas ajudam a organizar itens relacionados. Use ul quando a ordem não importa e ol quando a ordem importa. 

<ul class="cards-list">
  <li>Estrutura semântica</li>
  <li>Acessibilidade básica</li>
  <li>Responsividade preparada</li>
</ul>

<ol class="steps-list">
  <li>Criar HTML</li>
  <li>Aplicar classes</li>
  <li>Aprovar para CSS</li>
</ol>

🔗 Etapa 4 — Links profissionais

Links precisam dizer para onde levam. Evite textos genéricos como “clique aqui” quando o contexto não estiver claro. 

<a href="contato.html">Acessar página de contato</a>

<a href="https://github.com/usuario/projeto" target="_blank" rel="noopener noreferrer">
  Ver repositório no GitHub
</a>
Tipo de link Exemplo Cuidado
Interno href="contato.html" Use para páginas do mesmo projeto.
Âncora href="#sobre" O ID precisa existir na página.
Externo target="_blank" Use também rel="noopener noreferrer".

🖼️ Etapa 5 — Imagens com responsabilidade

Imagem não é decoração automática. Se a imagem comunica informação, ela precisa de alt descritivo. Se for puramente decorativa, pode usar alt="". 

<img
  src="assets/imagens/pescador.jpg"
  alt="Pescador em um barco durante o pôr do sol"
/>

🎥 Etapa 6 — Vídeo e áudio

Mídia precisa de controle, formato correto e contexto. O usuário precisa entender por que aquele vídeo ou áudio está ali. 

<video controls>
  <source src="assets/videos/apresentacao.mp4" type="video/mp4" />
  Seu navegador não suporta vídeo.
</video>

<audio controls>
  <source src="assets/audios/narracao.mp3" type="audio/mpeg" />
  Seu navegador não suporta áudio.
</audio>

🧭 Etapa 7 — Iframe com title

Todo iframe precisa de title. Isso ajuda acessibilidade e clareza semântica. 

<iframe
  src="https://www.windy.com/embed2.html"
  title="Mapa interativo de previsão do tempo"
></iframe>

✅ Checklist — Conteúdo HTML aprovado

  • O conteúdo tem ordem lógica
  • Títulos foram usados por hierarquia, não por tamanho visual
  • Parágrafos explicam ideias completas
  • Listas agrupam itens relacionados
  • Links têm texto claro
  • Links externos usam rel="noopener noreferrer"
  • Imagens informativas possuem alt
  • Vídeos e áudios têm contexto
  • Iframes possuem title
  • O conteúdo está preparado para receber CSS

⚠️ Erros proibidos

  • Usar h1 várias vezes sem critério.
  • Usar link com texto vago sem contexto.
  • Inserir imagem importante sem alt.
  • Colocar vídeo ou iframe sem explicar por que existe.
  • Usar lista apenas para criar visual, sem relação entre os itens.
  • Esquecer de fechar tags corretamente.

HTML5 na prática

📊 HTML5 — Tabelas e Formulários Profissionais

Tabelas e formulários exigem cuidado técnico. Tabela não é ferramenta de layout. Formulário não é apenas input jogado na tela. Ambos precisam de estrutura, clareza, acessibilidade e preparação para CSS.

Tabela organiza dados. Formulário coleta informações. Os dois precisam ser claros, acessíveis e fáceis de usar.

📊 Etapa 1 — Quando usar tabela

Tabela deve ser usada quando o conteúdo é realmente tabular: linhas e colunas com dados relacionados. Não uso tabela para montar layout visual.

  • O conteúdo tem linhas e colunas reais
  • Os dados precisam ser comparados
  • Existe cabeçalho para cada coluna
  • A tabela não está sendo usada para posicionar elementos

🧱 Etapa 2 — Estrutura correta de tabela

<div class="table-wrapper">
  <table>
    <thead>
      <tr>
        <th>Mês</th>
        <th>Evento</th>
        <th>Local</th>
      </tr>
    </thead>

    <tbody>
      <tr>
        <td>Janeiro</td>
        <td>Abertura da Temporada</td>
        <td>Rio Paraná</td>
      </tr>
    </tbody>
  </table>
</div>

A classe table-wrapper protege a tabela no celular. Quando a tabela for larga, ela pode rolar horizontalmente sem quebrar a página inteira.

✅ Regras profissionais para tabelas

Elemento Função Regra profissional
table Estrutura da tabela Use apenas para dados tabulares.
thead Cabeçalho da tabela Use para agrupar os títulos das colunas.
tbody Corpo da tabela Use para agrupar os dados principais.
tr Linha Cada linha representa um registro.
th Célula de cabeçalho Use para nomear colunas ou linhas.
td Célula de dados Use para valores comuns da tabela.

📝 Etapa 3 — Quando usar formulário

Formulário é usado quando o usuário precisa enviar informações: nome, email, mensagem, login, cadastro, busca, inscrição ou contato.

  • Existe uma ação clara para o usuário
  • Cada campo tem uma finalidade

🎨 CSS do Zero — Ligação, Variáveis, Reset e Base Global

Depois que o HTML foi aprovado, o CSS começa. A primeira missão não é sair colocando cor em tudo. A primeira missão é criar uma base visual organizada, previsível e preparada para crescer.

CSS profissional começa com sistema. Primeiro eu preparo a base, depois construo o layout.

🔗 Etapa 1 — Ligar o CSS no HTML

O arquivo CSS deve ser conectado dentro do head do HTML. Sem essa ligação, o navegador não aplica os estilos. 

<link rel="stylesheet" href="styles/style.css" />
  • Existe uma pasta styles
  • Existe um arquivo style.css
  • O caminho no href está correto
  • O CSS carrega no navegador

📁 Etapa 2 — Organizar o arquivo CSS

O arquivo CSS precisa ter ordem. Isso facilita manutenção, evita repetição e mostra maturidade técnica. 

/* 1. Tokens / variáveis */
/* 2. Reset */
/* 3. Base global */
/* 4. Acessibilidade */
/* 5. Layout */
/* 6. Componentes */
/* 7. Responsividade */

🎛️ Etapa 3 — Criar variáveis no :root

Variáveis guardam decisões importantes do projeto. Em vez de repetir cores e tamanhos em vários lugares, eu centralizo tudo no começo do CSS. 

:root {
  --color-bg: #050816;
  --color-surface: #111827;
  --color-text: #f8fafc;
  --color-muted: #94a3b8;
  --color-primary: #facc15;
  --color-border: rgba(148, 163, 184, 0.22);

  --font-main: system-ui, sans-serif;
  --container: 1180px;

  --space-sm: 1rem;
  --space-md: 1.5rem;
  --space-lg: 2.5rem;
  --space-xl: 4rem;

  --radius-md: 1rem;
  --radius-lg: 1.5rem;
}

🧹 Etapa 4 — Aplicar reset básico

O reset cria previsibilidade. Ele reduz diferenças entre navegadores e evita surpresas de espaçamento e tamanho. 

*,
*::before,
*::after {
  box-sizing: border-box;
}

body {
  margin: 0;
}

img,
video,
iframe {
  display: block;
  max-width: 100%;
}

🌍 Etapa 5 — Criar base global

A base global define a aparência inicial da página: fonte, cor, fundo, altura de linha, links e comportamento geral. 

html {
  scroll-behavior: smooth;
}

body {
  font-family: var(--font-main);
  line-height: 1.6;
  background: var(--color-bg);
  color: var(--color-text);
}

a {
  color: inherit;
}

button,
input,
textarea {
  font: inherit;
}

📐 CSS Layout — Container, Section, Grid e Flex

Depois que a base CSS está pronta, começa a construção do layout. Layout é a organização visual da página: largura, espaçamento, alinhamento, colunas, linhas, blocos e respiro.

O .container controla a largura do conteúdo. Ele impede que o texto fique grudado nas bordas no celular e largo demais no desktop.

  margin-inline: auto;

  • O conteúdo tem respiro lateral no celular

  • Não usei largura fixa desnecessária

📐 Etapa 3 — Usar Grid para estruturas maiores

Para projetos modernos, uma versão mais flexível pode usar minmax:

  align-items: center;
  gap: 1rem;

Seletores escolhem quais elementos serão estilizados. No meu padrão, eu priorizo classes claras em vez de depender de tags soltas.

📏 Valores e unidades

Uso rem, %, min(), clamp(), vw e vh com intenção, pensando em responsividade.

📦 Box Model

Todo elemento é uma caixa: conteúdo, padding, border e margin. Entender isso evita layout quebrado e espaçamento bagunçado.

🧬 Cascata e herança

O CSS segue ordem, especificidade e herança. Por isso, o arquivo precisa ser organizado por camadas e não escrito de qualquer jeito.

📱 Responsividade

O CSS precisa funcionar no celular, tablet, notebook e desktop. Um site bonito só no desktop ainda não é profissional.

⚙️ JavaScript do Zero — Ligação, DOM e Eventos

JavaScript entra depois que o HTML e o CSS foram aprovados. Ele adiciona comportamento, interação e resposta às ações do usuário. A primeira missão não é criar algo complexo: é ligar o arquivo corretamente, selecionar elementos com segurança e testar eventos.

JavaScript profissional não entra para consertar bagunça. Ele entra para dar vida a uma estrutura e interface que já foram aprovadas.

🧭 Etapa 1 — Confirmar que HTML e CSS estão aprovados

Antes de escrever JavaScript, eu confirmo se a estrutura e a interface estão prontas. Se o HTML está confuso ou o CSS quebra no celular, o JavaScript só aumenta o problema.

  • HTML foi aprovado no Portão Final HTML
  • CSS foi aprovado no Portão Final CSS
  • A página abre corretamente sem erro visual grave
  • O layout funciona no celular
  • A interface já está pronta para receber comportamento

🔗 Etapa 2 — Ligar o JavaScript no HTML

O JavaScript normalmente fica no final do body, porque assim o HTML já foi carregado antes do script tentar acessar os elementos. 

<script src="scripts/script.js"></script>
</body>
  • Existe uma pasta scripts
  • Existe um arquivo script.js
  • O caminho no src está correto
  • O script está ligado no final do body

🧠 Etapa 3 — Entender o DOM

DOM é a representação da página que o JavaScript consegue acessar e modificar. Com ele, eu posso selecionar elementos, trocar classes, ler valores, criar eventos e reagir ao usuário. 

const title = document.querySelector("h1");
console.log(title);

🎯 Etapa 4 — Selecionar elementos com segurança

Antes de usar um elemento, eu preciso garantir que ele existe. Isso evita erro no console e deixa o código mais seguro. 

const button = document.querySelector(".menu-button");

if (button) {
  console.log("Botão encontrado.");
}

🖱️ Etapa 5 — Criar evento de clique

Eventos permitem responder às ações do usuário. O exemplo mais comum é o clique em um botão. 

const button = document.querySelector(".menu-button");

if (button) {
  button.addEventListener("click", () => {
    console.log("Botão clicado.");
  });
}

🏷️ Etapa 6 — Usar classList

O classList permite adicionar, remover ou alternar classes. Isso é muito usado para abrir menus, mostrar mensagens, ativar estados e controlar visual. 

const button = document.querySelector(".menu-button");
const menu = document.querySelector(".site-nav");

if (button && menu) {
  button.addEventListener("click", () => {
    menu.classList.toggle("is-open");
  });
}

🧪 Etapa 7 — Testar no console

O console do navegador mostra erros, avisos e mensagens de teste. Antes de aprovar o JavaScript, eu preciso abrir o DevTools e conferir se não existe erro crítico.

1. Abrir DevTools Entrar na aba Console do navegador.
2. Recarregar a página Verificar se aparece erro vermelho ao carregar.
3. Testar botões e interações Clicar nos elementos e observar se tudo responde corretamente.
4. Corrigir antes de avançar Erro crítico no console bloqueia a próxima etapa.

✅ Checklist — Base JavaScript aprovada

  • JavaScript está ligado corretamente no HTML
  • O arquivo script.js existe
  • Usei querySelector com seletor claro
  • Conferi se o elemento existe antes de usar
  • Criei pelo menos um evento com intenção
  • Usei classList quando precisei alterar estado visual
  • Testei no console
  • Não existe erro crítico em vermelho
  • A interação funciona no celular e no desktop

⚠️ Erros proibidos nessa fase

  • Ligar o script no caminho errado.
  • Tentar usar elemento que não existe.
  • Ignorar erro vermelho no console.
  • Copiar código sem entender o que ele faz.
  • Usar JavaScript para esconder problema de HTML ou CSS.
  • Criar interação que só funciona no desktop.
  • Colocar lógica grande antes de dominar seleção e eventos.

📝 JavaScript — Formulários e Validações

Formulários conectam estrutura, visual e comportamento. O HTML cria os campos, o CSS organiza a experiência e o JavaScript melhora a validação, o feedback e a interação com o usuário.

Validação profissional não é só bloquear envio. É orientar o usuário com clareza, preservar acessibilidade e evitar erros antes da publicação.

🧭 Etapa 1 — Garantir que o formulário HTML está correto

Antes do JavaScript, o formulário precisa nascer bem no HTML: com form, label, input, name, id, type e botão claro. 

<form class="contact-form" action="#" method="post">
  <div class="form-group">
    <label for="nome">Nome</label>
    <input type="text" id="nome" name="nome" required />
  </div>

  <div class="form-group">
    <label for="email">Email</label>
    <input type="email" id="email" name="email" required />
  </div>

  <button type="submit">Enviar mensagem</button>
</form>

🎯 Etapa 2 — Capturar o formulário

Primeiro eu seleciono o formulário com uma classe clara. Depois verifico se ele existe antes de criar qualquer evento. 

const form = document.querySelector(".contact-form");

if (form) {
  console.log("Formulário encontrado.");
}

📨 Etapa 3 — Escutar o envio com submit

O evento submit acontece quando o usuário tenta enviar o formulário. 

const form = document.querySelector(".contact-form");

if (form) {
  form.addEventListener("submit", (event) => {
    event.preventDefault();

    console.log("Formulário enviado.");
  });
}

⛔ Etapa 4 — Usar preventDefault com intenção

O preventDefault() impede o comportamento padrão do navegador. Em estudos e projetos sem back-end, ele evita que a página recarregue antes da validação. 

event.preventDefault();

📥 Etapa 5 — Ler valores dos campos

Depois de capturar o formulário, eu leio os campos pelo id ou por seletores internos. 

const nameInput = form.querySelector("#nome");
const emailInput = form.querySelector("#email");

const nameValue = nameInput.value.trim();
const emailValue = emailInput.value.trim();

✅ Etapa 6 — Validar campos vazios

Mesmo usando required no HTML, o JavaScript pode melhorar o feedback e organizar a lógica. 

if (nameValue === "") {
  alert("Digite seu nome.");
  return;
}

📧 Etapa 7 — Validar email simples

Uma validação inicial de email pode verificar se existe @ e ponto. Para projetos maiores, posso usar regras mais completas. 

if (!emailValue.includes("@") || !emailValue.includes(".")) {
  alert("Digite um email válido.");
  return;
}

💬 Etapa 8 — Mostrar mensagem para o usuário

O usuário precisa entender o que aconteceu. Em projetos simples, posso usar alert. Em projetos mais profissionais, posso exibir uma mensagem dentro da página. 

alert("Mensagem enviada com sucesso!");

🧩 Exemplo completo de validação simples

const form = document.querySelector(".contact-form");

if (form) {
  form.addEventListener("submit", (event) => {
    event.preventDefault();

    const nameInput = form.querySelector("#nome");
    const emailInput = form.querySelector("#email");

    const nameValue = nameInput.value.trim();
    const emailValue = emailInput.value.trim();

    if (nameValue === "") {
      alert("Digite seu nome.");
      return;
    }

    if (!emailValue.includes("@") || !emailValue.includes(".")) {
      alert("Digite um email válido.");
      return;
    }

    alert("Mensagem enviada com sucesso!");
    form.reset();
  });
}

♿ Etapa 9 — Preservar acessibilidade

Validação precisa ajudar, não atrapalhar. Mensagens devem ser claras, campos precisam manter labels e o usuário deve entender o erro.

  • O formulário tem labels visíveis
  • Os campos têm tipos corretos
  • A mensagem de erro é clara
  • O usuário entende como corrigir
  • O formulário continua usável no celular

✅ Checklist — Formulário JS aprovado

  • O formulário foi selecionado com segurança
  • O evento submit foi criado
  • Usei preventDefault() com intenção
  • Li os valores dos campos corretamente
  • Usei trim() para limpar espaços
  • Validei campo vazio
  • Validei email básico
  • Mostrei mensagem clara para o usuário
  • Testei envio com campos vazios
  • Testei envio com email inválido
  • Testei envio correto
  • Console está sem erro crítico

⚠️ Erros proibidos em validações

  • Validar formulário sem label nos campos.
  • Não testar campo vazio.
  • Não testar email inválido.
  • Deixar erro vermelho no console.
  • Mostrar mensagem confusa para o usuário.
  • Usar JavaScript sem entender o fluxo do submit.
  • Apagar dados do formulário antes de validar.

✨ JavaScript — Interações Reais da Página

Depois de dominar ligação, DOM, eventos e formulários, o JavaScript começa a criar interações reais: abrir menu, voltar ao topo, ativar links, mostrar mensagens, salvar preferências e melhorar a experiência do usuário.

Interação profissional precisa ter propósito. Ela deve melhorar a navegação, o feedback ou a experiência sem prejudicar acessibilidade e desempenho.

🔝 Etapa 1 — Botão voltar ao topo

O botão voltar ao topo ajuda em páginas longas. Ele aparece quando o usuário rola a página e leva de volta ao início. 

const backToTopButton = document.querySelector(".back-to-top");

if (backToTopButton) {
  window.addEventListener("scroll", () => {
    if (window.scrollY > 700) {
      backToTopButton.classList.add("is-visible");
    } else {
      backToTopButton.classList.remove("is-visible");
    }
  });

  backToTopButton.addEventListener("click", () => {
    window.scrollTo({
      top: 0,
      behavior: "smooth",
    });
  });
}

🧭 Etapa 2 — Menu abrir e fechar

Em telas menores, menus podem ser recolhidos. O JavaScript pode alternar uma classe para abrir ou fechar a navegação. 

const menuButton = document.querySelector(".menu-button");
const menu = document.querySelector(".site-nav");

if (menuButton && menu) {
  menuButton.addEventListener("click", () => {
    menu.classList.toggle("is-open");
  });
}

🏷️ Etapa 3 — Trocar classe ativa

Classes ativas ajudam a mostrar qual seção, botão ou item está selecionado. 

const navLinks = document.querySelectorAll(".site-nav a");

navLinks.forEach((link) => {
  link.addEventListener("click", () => {
    navLinks.forEach((item) => item.classList.remove("is-active"));
    link.classList.add("is-active");
  });
});

👁️ Etapa 4 — Mostrar e esconder elementos

Mostrar e esconder elementos é útil para mensagens, perguntas frequentes, modais simples e detalhes extras. 

const toggleButton = document.querySelector(".toggle-button");
const panel = document.querySelector(".details-panel");

if (toggleButton && panel) {
  toggleButton.addEventListener("click", () => {
    panel.classList.toggle("is-open");
  });
}

💬 Etapa 5 — Mensagem visual para o usuário

Em vez de depender sempre de alert, projetos mais profissionais podem mostrar uma mensagem dentro da página. 

const message = document.querySelector(".form-message");

function showMessage(text) {
  if (!message) return;

  message.textContent = text;
  message.classList.add("is-visible");
}

showMessage("Mensagem enviada com sucesso!");

💾 Etapa 6 — Salvar dados simples no localStorage

O localStorage permite salvar informações simples no navegador, como checklists marcados, tema escolhido ou preferências do usuário. 

localStorage.setItem("tema", "escuro");

const savedTheme = localStorage.getItem("tema");

console.log(savedTheme);

✅ Exemplo — Checklist salvo

const checks = document.querySelectorAll(".checklist input");

checks.forEach((input, index) => {
  const savedValue = localStorage.getItem(`check-${index}`);

  if (savedValue === "true") {
    input.checked = true;
  }

  input.addEventListener("change", () => {
    localStorage.setItem(`check-${index}`, input.checked);
  });
});

🧭 Como Escolher a Estrutura Certa

Antes de começar um projeto, eu preciso identificar que tipo de entrega estou construindo. Isso evita improviso, excesso de código, seções desnecessárias e escolhas ruins de HTML, CSS e JavaScript.

Primeiro eu entendo o tipo de projeto. Depois escolho a estrutura. Depois construo com método.

⚡ Decisão rápida

Uma página para apresentar empresa, negócio ou serviço Site institucional Clareza, confiança, serviços e contato
Uma página para convencer alguém a tomar uma ação Landing page Proposta de valor, benefícios e CTA
Uma tela de entrada com email e senha Tela de login Formulário, acessibilidade e validação
Uma página para me apresentar como dev Portfólio pessoal Projetos, habilidades, links e contato

Uma página para vender ou apresentar produto Página de produto Imagem, descrição, benefícios e CTA

Um projeto final de curso Projeto completo HTML/CSS/JS Construção, validação, publicação e portfólio

🧠 Perguntas antes de escolher

  • O projeto é informativo, comercial, interativo ou de carreira?
  • Ele precisa de formulário?
  • Ele precisa de JavaScript agora?
  • Ele precisa ser publicado no portfólio?
  • Ele precisa de imagens, tabelas, cards ou dashboard?
  • O usuário precisa clicar, enviar, filtrar, salvar ou navegar?

🏗️ Caminho por tipo de projeto

🌐 Projeto de apresentação

Use quando o objetivo é mostrar uma empresa, pessoa, serviço, produto ou projeto. Comece com hero, sobre, benefícios, cards e contato.

📝 Projeto com formulário

Use quando o usuário precisa enviar dados. Priorize labels, inputs corretos, acessibilidade, validação e feedback claro.

⚙️ Projeto com JavaScript

Use quando existe interação real: adicionar, remover, filtrar, salvar, abrir, fechar, validar ou alterar estados da interface.

💼 Projeto para carreira

Use quando o objetivo é apresentar sua evolução. Precisa de README, link publicado, prints, explicação técnica e evidência para recrutador.

🚦 Sinais de que escolhi a estrutura errada

  • Estou criando seções sem saber por quê.
  • Estou usando tabela para layout visual.
  • Estou usando JavaScript sem necessidade real.
  • Estou fazendo uma landing page sem chamada para ação.
  • Estou fazendo formulário sem label e sem validação.
  • Estou criando portfólio sem links, projetos e explicação.
  • Estou começando pelo visual antes de entender a estrutura.

✅ Checklist — Estrutura escolhida corretamente

  • Identifiquei o objetivo do projeto
  • Escolhi o tipo de estrutura mais adequado
  • Listei as seções necessárias
  • Defini se haverá formulário, tabela, cards ou mídia
  • Defini se JavaScript será necessário
  • Sei qual evidência esse projeto vai gerar
