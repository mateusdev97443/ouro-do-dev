Sistema de trabalho

🏗️ Fluxo de Construção Profissional

Esta é a espinha dorsal do Ouro do Dev. Todo projeto deve seguir uma ordem profissional: primeiro a estrutura, depois a aparência, depois o comportamento, depois a validação, publicação e apresentação.

Nenhuma etapa avança sem aprovação da etapa anterior. Primeiro eu construo certo, depois eu evoluo com segurança.
1. Planejamento da página Definir objetivo, público, conteúdo, páginas, seções e função do projeto.
2. HTML — Construção da estrutura Criar o esqueleto semântico com head, header, nav, main, sections, articles, aside e footer.
3. Portão de aprovação HTML Validar estrutura, semântica, acessibilidade básica, organização e prontidão para CSS.
4. CSS — Construção visual Aplicar variáveis, reset, base global, layout, componentes, responsividade e acabamento visual.
5. Portão de aprovação CSS Validar visual, responsividade, contraste, proporção, leitura, DevTools e qualidade da interface.
6. JavaScript — Comportamento Adicionar interações, eventos, DOM, validações, menus, filtros, busca e funcionalidades reais.
7. Portão de aprovação JavaScript Conferir console, eventos, acessibilidade, funcionamento no celular e ausência de erros críticos.
8. GitHub e publicação Fazer commits claros, README, GitHub Pages, link publicado e organização do repositório.
9. Portfólio e carreira Salvar prints, registrar aprendizados, explicar decisões e transformar o projeto em evidência profissional.

🧱 Portão de aprovação HTML

O HTML só está pronto para receber CSS quando sua estrutura está clara, semântica, acessível e organizada. CSS não deve ser usado para esconder HTML mal construído.

  • O documento possui <!doctype html>
  • O idioma está definido com lang="pt-BR"
  • O meta viewport foi configurado
  • O title descreve corretamente o projeto
  • A meta description existe e faz sentido
  • A página possui header, main e footer
  • As seções têm ordem lógica e títulos claros
  • Imagens informativas possuem alt
  • Formulários possuem label conectado aos campos
  • Tabelas são usadas apenas para dados reais
  • Iframes possuem title
  • O HTML está indentado e legível

🎨 Portão de aprovação CSS

O CSS só está pronto quando a interface está organizada, confortável, responsiva, acessível e coerente com o objetivo do projeto.

  • O CSS está ligado corretamente no HTML
  • Foram criadas variáveis no :root
  • O reset básico foi aplicado
  • O layout usa containers e espaçamentos consistentes
  • O site foi testado em celular
  • O site foi testado em desktop
  • Cards, botões, tabelas e formulários estão confortáveis
  • O contraste está legível
  • O foco visível foi preservado
  • A tabela não estoura a tela no celular
  • O DevTools foi usado para revisar layout
  • O visual passa organização e profissionalismo

⚙️ Portão de aprovação JavaScript

JavaScript entra para adicionar comportamento, não para bagunçar a estrutura. Uma função só deve existir quando melhora a experiência, validação ou interação do projeto.

  • O JavaScript está ligado no final do HTML
  • O console não possui erro crítico
  • Botões e eventos funcionam corretamente
  • As interações funcionam no celular
  • O JS não prejudica acessibilidade
  • O código está separado em funções claras
  • O comportamento esperado foi testado manualmente

🌐 Portão final de publicação

Um projeto só vira evidência profissional quando está publicado, documentado, testado e explicável.

  • O projeto possui README completo
  • O GitHub está organizado
  • Os commits têm mensagens claras
  • O GitHub Pages ou deploy foi testado

Portão de aprovação

✅ HTML Pronto para CSS

Antes de começar a estilizar uma página, eu preciso garantir que o HTML está preparado. Um HTML profissional não é apenas um conjunto de tags funcionando. Ele precisa ter estrutura, semântica, organização, acessibilidade básica e classes estratégicas.

CSS não deve corrigir HTML mal planejado. Primeiro a estrutura fica certa. Depois a aparência entra com segurança.

🧭 Etapa 1 — Entender o objetivo da página

Antes de escrever código, eu preciso saber o que a página deve comunicar. Isso evita criar tags aleatórias, seções sem sentido e conteúdo desorganizado.

  • Defini o tema do projeto
  • Defini o público da página
  • Defini o objetivo principal
  • Listei as seções necessárias
  • Sei qual ação o usuário deve fazer

🧱 Etapa 2 — Criar a base obrigatória do documento

Todo projeto começa com uma base HTML limpa. Essa estrutura garante idioma correto, responsividade, título, descrição e conexão com o CSS. 

<!doctype html>
<html lang="pt-BR">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Nome do Projeto</title>
    <meta name="description" content="Descrição clara do projeto." />
    <link rel="stylesheet" href="styles/style.css" />
  </head>

  <body>
  </body>
</html>

🏛️ Etapa 3 — Organizar o corpo da página

O body precisa ter uma arquitetura clara. O caminho profissional começa com header, main e footer. Depois entram as seções internas. 

<body>
  <header class="site-header">
    <nav class="site-nav" aria-label="Navegação principal">
      <!-- links principais -->
    </nav>
  </header>

  <main class="page-main">
    <section class="hero"></section>
    <section class="section"></section>
    <article class="section"></article>
    <aside class="section"></aside>
  </main>

  <footer class="site-footer"></footer>
</body>

🏷️ Etapa 4 — Aplicar classes estratégicas

As classes entram antes do CSS para preparar o projeto. Elas precisam explicar o papel do bloco, não apenas a aparência dele.

Parte Classe recomendada Motivo
Cabeçalho site-header Identifica o topo principal do site.
Navegação site-nav Separa claramente a área de links.
Conteúdo principal page-main Marca a área central da página.
Primeira seção hero Define a primeira dobra de destaque.
Seções comuns section Cria padrão para blocos repetidos.
Cards card Prepara blocos visuais reutilizáveis.
Tabela table-wrapper Protege a tabela no celular.
Formulário form-group Agrupa label, campo e mensagem.
Rodapé site-footer Identifica o fechamento da página.

♿ Etapa 5 — Revisar acessibilidade básica antes do CSS

Acessibilidade não começa no CSS. Ela começa no HTML. Se a estrutura já nasce errada, a página fica fraca mesmo que o visual fique bonito.

  • Navegação possui aria-label quando necessário
  • Imagens informativas possuem alt
  • Inputs possuem label conectado com for e id
  • Botões têm texto claro
  • Links dizem para onde levam
  • Iframes possuem title
  • A hierarquia de títulos faz sentido

🧪 Etapa 6 — Validar se o HTML está pronto

Antes de ir para o CSS, eu faço uma revisão completa. Essa aprovação protege o projeto contra retrabalho e cria base sólida para o visual.

1. Ler o HTML de cima para baixo Verificar se a estrutura tem lógica: topo, conteúdo principal e rodapé.
2. Conferir semântica Garantir que cada tag foi usada pelo significado, não por aparência.
3. Conferir acessibilidade Revisar alt, labels, title, links, headings e navegação.
4. Conferir classes Verificar se as classes são claras, reutilizáveis e preparadas para CSS.
5. Abrir no navegador Mesmo sem CSS, a página precisa mostrar conteúdo em ordem compreensível.

✅ Checklist final — HTML aprovado para CSS

  • A estrutura base do documento está completa
  • O head possui charset, viewport, title, description e link CSS
  • O body está organizado em blocos semânticos
  • Existe apenas um main principal
  • As seções têm títulos claros
  • As classes estratégicas foram aplicadas
  • As imagens possuem alt adequado
  • Os formulários possuem label
  • Tabelas possuem estrutura correta
  • Iframes possuem title
  • O HTML está indentado
  • A página aberta sem CSS ainda faz sentido

Fundação profissional

🧱 HTML Mestre — Base de Todo Projeto Front-End

HTML é a estrutura da página. Antes de pensar em cor, animação, layout ou JavaScript, eu preciso criar uma base semântica, acessível, organizada e fácil de manter.

HTML profissional não é só fazer aparecer na tela. É estruturar informação com sentido, acessibilidade e intenção.

✅ Estrutura

Todo projeto começa com doctype, idioma correto, metadados, título, descrição, CSS ligado e conteúdo dentro de body.

✅ Semântica

Tags como header, main, section, article, aside e footer dão significado ao conteúdo.

✅ Acessibilidade

Imagens precisam de alt, formulários precisam de label, iframes precisam de title e a navegação precisa fazer sentido.

✅ Organização

O HTML deve ser legível, indentado, separado por blocos e preparado para receber classes estratégicas.

✅ SEO básico

Título claro, meta description, hierarquia de headings e conteúdo bem estruturado ajudam mecanismos de busca e usuários.

✅ Manutenção

Outro desenvolvedor precisa bater o olho no HTML e entender a página sem sofrer.

📌 Esqueleto HTML obrigatório

<!doctype html>
<html lang="pt-BR">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Nome do Projeto</title>
    <meta name="description" content="Descrição clara do projeto." />
    <link rel="stylesheet" href="styles/style.css" />
  </head>

  <body>
    <header class="site-header">
      <nav class="site-nav" aria-label="Navegação principal">
        <!-- links -->
      </nav>
    </header>

    <main class="page-main">
      <section class="hero"></section>
      <section class="section"></section>
    </main>

    <footer class="site-footer"></footer>

    <script src="scripts/script.js"></script>
  </body>
</html>

🧭 Tags essenciais e função profissional

Tag Função Regra profissional
header Cabeçalho Use para topo da página ou introdução de uma seção importante.
nav Navegação Use com aria-label quando houver menu de links.
main Conteúdo principal Deve existir apenas um main principal por página.
section Seção temática Use quando o bloco tiver um assunto próprio e geralmente um título.
article Conteúdo independente Use para post, card completo, notícia, aula, projeto ou item autônomo.
aside Conteúdo complementar Use para dicas, curiosidades, observações e conteúdo lateral.
footer Rodapé Use para fechamento, direitos, links finais e informações extras.
form Formulário Todo campo precisa de label ligado ao id.
table Tabela Use para dados tabulares, não para layout visual.
img Imagem Imagem informativa precisa de alt descritivo.
iframe Conteúdo incorporado Sempre usar title claro e descritivo.

  • Criei meta description
  • Usei apenas um main principal
  • Organizei o conteúdo com tags semânticas

  • Formulários têm label ligado ao campo
  • Iframes têm title
  • Tabelas foram usadas apenas para dados
  • Código está indentado e legível

⚠️ Erros proibidos em HTML

  • Usar várias div sem necessidade.
  • Colocar tudo dentro de uma única seção gigante.
  • Usar heading fora de ordem sem motivo.
  • Imagem importante sem alt.
  • Input sem label.
  • Iframe sem title.
  • Tabela usada para montar layout.
  • HTML sem indentação.
  • Link com texto genérico como “clique aqui” sem contexto.

Portão de aprovação

🎨 CSS Pronto para JavaScript

Depois que o HTML está aprovado, o CSS entra para transformar a estrutura em interface. Essa etapa não é apenas “deixar bonito”. É criar um sistema visual organizado, responsivo, acessível, confortável e preparado para receber comportamento com JavaScript.

JavaScript não deve entrar para consertar interface mal construída. Primeiro o visual precisa estar organizado, responsivo e aprovado.

🧭 Etapa 1 — Confirmar que o HTML está aprovado

Antes de escrever CSS, eu volto ao portão anterior e confirmo se a estrutura está pronta. Se o HTML ainda está confuso, o CSS vira gambiarra.

  • O HTML tem estrutura base completa
  • O conteúdo está organizado em blocos semânticos
  • As classes estratégicas já foram aplicadas
  • Acessibilidade básica foi revisada
  • A página sem CSS ainda faz sentido

🎛️ Etapa 2 — Criar o sistema de variáveis

Variáveis tornam o CSS mais profissional porque centralizam decisões visuais: cores, fontes, espaçamentos, sombras, bordas e tamanhos. 

:root {
  --color-bg: #050816;
  --color-surface: #111827;
  --color-text: #f8fafc;
  --color-muted: #94a3b8;
  --color-primary: #facc15;
  --color-border: rgba(148, 163, 184, 0.22);

  --font-main: system-ui, sans-serif;
  --container: 1180px;

  --space-sm: 1rem;
  --space-md: 1.5rem;
  --space-lg: 2.5rem;
  --space-xl: 4rem;

  --radius-md: 1rem;
  --radius-lg: 1.5rem;
}

🧹 Etapa 3 — Aplicar reset e base global

O reset remove comportamentos inconsistentes do navegador e cria uma base previsível. A base global define leitura, cor, fonte e comportamento geral da página. 

*,
*::before,
*::after {
  box-sizing: border-box;
}

html {
  scroll-behavior: smooth;
}

body {
  margin: 0;
  font-family: var(--font-main);
  line-height: 1.6;
  background: var(--color-bg);
  color: var(--color-text);
}

img,
video,
iframe {
  display: block;
  max-width: 100%;
}

📐 Etapa 4 — Criar layout base

O layout profissional começa com largura controlada, espaçamentos consistentes e seções previsíveis. 

.container {
  width: min(100% - 2rem, var(--container));
  margin-inline: auto;
}

.section {
  padding-block: var(--space-xl);
}

.grid {
  display: grid;
  gap: var(--space-md);
}

🧱 Etapa 5 — Estilizar componentes principais

Componentes são partes reutilizáveis da interface. Um projeto profissional não estiliza tudo de forma aleatória. Ele cria padrões.

Componente O que revisar Critério profissional
Header Logo, navegação e alinhamento Precisa orientar o usuário sem poluir a tela.
Hero Título, texto, botão e impacto inicial Precisa deixar claro o objetivo da página.
Cards Espaçamento, borda, sombra e leitura Precisam parecer parte do mesmo sistema visual.
Botões Tamanho, contraste, toque e foco Devem ser claros, acessíveis e confortáveis no celular.
Formulários Labels, campos, espaçamento e botão Devem ser fáceis de preencher e entender.
Tabelas Largura, rolagem e legibilidade Não podem estourar a tela no celular.

📱 Etapa 6 — Testar responsividade

CSS só pode ser aprovado se funcionar em diferentes telas. Um layout profissional precisa se adaptar sem perder leitura, organização e conforto.

  • 320px — celular pequeno aprovado
  • 375px — celular comum aprovado
  • 425px — celular grande aprovado
  • 768px — tablet aprovado
  • 1366px — notebook aprovado
  • 1920px — desktop grande aprovado

🔍 Etapa 7 — Inspecionar com DevTools

DevTools é parte do processo. Eu não aprovo CSS apenas olhando de longe. Eu inspeciono estrutura, espaçamento, regras aplicadas, grid, flex, largura e comportamento.

1. Selecionar o bloco Inspecionar header, hero, cards, tabela, formulário e footer.
2. Conferir box model Ver margin, padding, border e tamanho real do elemento.
3. Testar valores Ajustar temporariamente no DevTools antes de salvar no CSS.
4. Validar responsivo Usar modo dispositivo para conferir celular e tablet.
5. Aplicar no arquivo Salvar no CSS apenas o que foi aprovado visualmente.

  • O arquivo CSS tem ordem lógica
  • Variáveis foram usadas para decisões importantes

  • Container e seções estão consistentes
  • Header e navegação estão claros
  • Hero comunica bem o objetivo da página
  • Cards seguem o mesmo padrão visual

  • Usar JS para esconder problema de CSS.

  • Não usar DevTools antes de aprovar.

Sistema visual profissional

🎨 CSS ME Standard — Meu Padrão Oficial de CSS

O CSS ME Standard é o meu padrão pessoal para transformar HTML semântico em interfaces modernas, organizadas, responsivas, acessíveis e fáceis de manter.

CSS profissional não é sair colocando cor. É criar um sistema visual com ordem, intenção, consistência e capacidade de manutenção.

🧠 O que é CSS

CSS é a camada visual da página. Ele controla cores, tamanhos, espaçamentos, fontes, layout, responsividade, estados e aparência dos componentes.

🏷️ Seletores

Seletores escolhem quais elementos serão estilizados. No meu padrão, eu priorizo classes claras em vez de depender de tags soltas.

📏 Valores e unidades

Uso rem, %, min(), clamp(), vw e vh com intenção, pensando em responsividade.

📦 Box Model

Todo elemento é uma caixa: conteúdo, padding, border e margin. Entender isso evita layout quebrado e espaçamento bagunçado.

🧬 Cascata e herança

O CSS segue ordem, especificidade e herança. Por isso, o arquivo precisa ser organizado por camadas e não escrito de qualquer jeito.

📱 Responsividade

O CSS precisa funcionar no celular, tablet, notebook e desktop. Um site bonito só no desktop ainda não é profissional.

1. Inspecionar o elemento Clicar com botão direito, abrir DevTools e selecionar o bloco visual.
2. Ver o CSS aplicado Conferir quais regras estão ativas e de qual linha elas vêm.

4. Testar alterações temporárias Editar valores no DevTools antes de mexer no arquivo real.
5. Aplicar no CSS oficial Só depois de validar visualmente, salvar a mudança no arquivo.

⚠️ Erros proibidos em CSS

  • Escrever CSS sem ordem.
  • Usar nomes como .caixa1, .texto2 e .div3.
  • Repetir a mesma cor em vários lugares sem variável.
  • Colocar todo estilo direto em tags sem necessidade.
  • Ignorar responsividade.
  • Remover foco visível sem substituir por outro.
  • Usar margem aleatória para “empurrar” layout.
  • Não testar no celular.
  • Não usar DevTools para entender o problema.

⚙️ JavaScript Pronto para Publicação

JavaScript entra depois que o HTML e o CSS já estão aprovados. Ele serve para adicionar comportamento, interação, validação e funcionalidades reais. Ele não deve ser usado para consertar uma estrutura ruim ou esconder uma interface mal resolvida.

JavaScript profissional melhora a experiência sem quebrar estrutura, acessibilidade, responsividade ou clareza do projeto.

🧭 Etapa 1 — Confirmar que HTML e CSS estão aprovados

Antes de escrever JavaScript, eu reviso os portões anteriores. Se o layout ainda quebra no celular ou o HTML está mal estruturado, o JavaScript vai aumentar a bagunça.

  • HTML foi aprovado como estrutura profissional
  • CSS foi aprovado como interface responsiva
  • A página funciona visualmente sem JavaScript
  • O layout está confortável no celular
  • Acessibilidade básica foi preservada

🔗 Etapa 2 — Ligar o JavaScript corretamente

O arquivo JavaScript normalmente deve ser ligado no final do body, depois do HTML principal. Assim o navegador já carregou os elementos antes do script tentar acessá-los. 

<script src="scripts/script.js"></script>
</body>

🎯 Etapa 3 — Selecionar elementos com segurança

Antes de criar eventos, eu preciso selecionar os elementos certos e garantir que eles existem na página. Isso evita erros no console. 

const button = document.querySelector(".menu-button");
const menu = document.querySelector(".site-nav");

if (button && menu) {
  button.addEventListener("click", () => {
    menu.classList.toggle("is-open");
  });
}

🖱️ Etapa 4 — Criar eventos com intenção

Eventos são respostas a ações do usuário: clique, digitação, envio de formulário, rolagem, foco ou mudança de campo.

Evento Uso comum Cuidado profissional
click Botões, menus e ações O elemento precisa ter nome claro e ser acessível.
submit Formulários Validar dados sem impedir acessibilidade.
input Campos de texto Evitar validações agressivas que atrapalham digitação.
change Checkbox, select e opções Salvar ou reagir apenas quando fizer sentido.
scroll Menu ativo, botão voltar ao topo Evitar excesso de lógica pesada na rolagem.

🧪 Etapa 5 — Validar formulários com responsabilidade

JavaScript pode melhorar formulários, mas o HTML já deve começar com atributos básicos como required, type="email", minlength e label. 

const form = document.querySelector(".contact-form");

if (form) {
  form.addEventListener("submit", (event) => {
    event.preventDefault();

    const email = form.querySelector("#email");

    if (!email.value.includes("@")) {
      alert("Digite um email válido.");
      return;
    }

    alert("Formulário enviado com sucesso!");
  });
}

♿ Etapa 6 — Preservar acessibilidade

JavaScript não pode tornar a página pior para quem usa teclado, leitor de tela ou navegação assistiva. Interação profissional precisa ser clara, previsível e acessível.

  • Botões interativos usam button, não div
  • Elementos clicáveis possuem texto claro
  • Menu abre e fecha sem perder navegação
  • Estados visuais não dependem apenas de cor
  • Foco visível continua funcionando
  • O usuário entende o que aconteceu após a ação

🔍 Etapa 7 — Conferir o console

Antes de publicar, eu abro o DevTools e verifico o console. Um projeto profissional não deve ser entregue com erro crítico de JavaScript.

1. Abrir DevTools Entrar na aba Console do navegador.
2. Recarregar a página Verificar se aparece erro vermelho logo ao carregar.
3. Testar interações Clicar em botões, menus, checklists, formulários e elementos dinâmicos.
4. Corrigir erros críticos Erro vermelho no console precisa ser entendido antes da publicação.
5. Testar no celular Interação que funciona no desktop também precisa funcionar no toque.

⚠️ Erros proibidos antes de publicar

  • Publicar projeto com erro vermelho no console.
  • Usar JavaScript para compensar HTML mal estruturado.
  • Usar JavaScript para esconder CSS quebrado.
  • Criar interação que só funciona no desktop.
  • Transformar div em botão sem necessidade.
  • Remover foco ou dificultar navegação por teclado.
  • Não testar formulário antes da entrega.
  • Copiar script sem entender o que ele faz.
